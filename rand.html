Using Git and Github

Sat 03 January 2015

Why Git?

is one of the most important tool in software development right now. The advent of websites like github has contributed greately to making the life of a developer whole lot easier. To understand why some sort of version control is a necessity of our time, let me try to think of a scenario.

Imagine how we wrote code the first time when we were learning things. They were messy and that's alright since we were just bothered about getting to see something work. The happiness of seeing something work properly is short lived, because soon you would feel like you could improve upon this somehow. So this is the sequence of things that generally happen.

  1. You create some stuff X. It works and you are super happy.

  2. Suddenly you see a possible improvement that could be made. So you would play around with X for a while and there are two possible outcomes here.

  3. We now have a better X or we are in the territory that Forrest Gump rightly said 'Shit Happens'

How do we tackle this issue?

We had working code which went all crazy after a small change that you made somewhere that you don't remember, I have heard stories of people waking up all night and realizing finally that their code is not working for trivial things like a misplaced '/' or ';' . The programmer who wasted 6 hours on fixing a ';' will surely thing of a better way to go about things.

Enter

We are not quite there. At first there is this mad thing we do with folders

  1. We create X , it is working fine and we put it in FOLDER_V1

  2. For every modification or new feature we add to X ,we create 'n' number of such folders ranging from FOLDER_V1 to FOLDER_V100 . Isn't this crazy or what ?

Solution ?

Git or any other version control saves you from the stink of having different versions of your code in separate folders. It tracks each of the change that we make and gives us control and peace over our code. It does a lot of cool other things and how Git does all this is probably beyond the scope of this writeup, a good place to start will be here

Time to play with Git

Install Git

If you are running a linux machine

sudo apt-get install git

If you are more of a GUI person and is new to git and has a windows machine , Github for Windows might be a good choice for working. But i like the feel of working in command line mode.

Create an account in Github

It is imperative that we have our code backed up somewhere other than our local machine. Github is a great place to do this and many other things . Do it

Creating a local repository

Head over to the directory where your source code resides.

git init

This creates a new local git repository. Git now magically monitors your directory and will know each and every change you will make in the directory.

Creating a remote repository

Github gives us the ability to have all our code stored remotely . This gives us many options like having multiple collabarators who work together to improve our code. Also this saves us from scenarios like

All my code was in local machine and it crashed

Well you have no reason to worry if you have a remote repostory

I skipped a step here, you would have to create a repository in github first. Upon creating a repository, you will get a git url which looks like this

https://github.com/Your_Github_Username/Your_repository_name.git

git add remote origin <'Your Remote repository git url'>

We are essentially telling git this

This is the remote repository where we would want to sync our local repository with.

origin is a name which we are giving here to refer to this remote repository. You could alternatively give any other name based on your preference. It is standard practice to name the repository as origin

Add files that matter to us

We can either add files individually like this

git add <'filename'>

or we can add all the files in a directory at one go.

git add --all

Commiting the changes

Remember from our SQL days that unless we perform the commit operation, changes to database wont be permanent . In parlance commit means something similar, it is the way by which git knows you have made changes to the directory that needs to be tracked. In essence we are saying to git

we have made some changes to the directory and this changes need to be tracked 

git commit -m '<YOUR-COMMIT-MESAGE>'

The commit message is used to describe the change that we had made . It could be something like "fixed the issue with IE8" .

Pushing the changes to Github

We have asked git in our local machine to keep track of changes. But we want to do the same with our remote git repository which is github. Or in other words we have to sync our local repository with remote repository. 

git push origin master

origin is the name we had given to our repository for referring. We are pushing this repository to a branch called master.

Think of branches as a seperate container where our code resides. It is intutive to name our main copy of working code as master and is hence the standard practice

A bit more on Branches

Suppose we are trying to add something new to our existing codebase. Think of this as a new feature that we are testing out or believe we can incorporate to our codebase. The smart thing to do is work on it seperately from our master branch. So what we do is, copy our master branch to a new branch and tell git to specifically that we are working on the new branch. This way git ensures that our code in master branch remains isolated from the new branch we are creating.

Here is how we create a new branch

git checkout -b <'new_branch_name'>

You might have heard of checkout already. We are essentially creating a new branch by copying the content of master branch here after this.

We need to tell git specifically to point to a particular branch.

git checkout master

This tells git that now we are working on master branch and changes for this branch needs to be tracked.

Knowing which branch we are working on.

This will show which branch git is currently pointing to.

git status

We will see response like "On master branch"

Don't forget to push the new branch to remote repo

We have finished working on the new branch and it can be pushed to remote repository by this

git push origin <'new_branch_name'>

Now if we go and check in github, we could see the seperate branches created.

Merging the new feature to our master copy

We always try to keep one master copy which we would want to keep as our perfect codebase or something we want to keep isolated. But now our new feature has been well and truly tested and we are ready to make a new master copy by integrating the changes present in our new branch.

Fist we point git to our master branch using checkout

git checkout master

Then we can merge the new branch to master 

git merge <'new_branch_we_created'>

Git will merge the new branch we created with the master branch unless there are any conflicts. If there are any conflicts we would have to make the changes manually and then try to merge again.

Changing my local machine, what do I do?

We already have pushed our code to Github and get it in our new machine pretty easily.

  1. Create a new local repo using git init

  2. Sync our local repo with the remote repo git remote add origin <'our_remote_repo_git_url'>

Now we can pull the latest version of our remote repository.

git pull origin <'branch_name'>

We can also simply give git pull and the branch that Github considers as the default branch will be pulled, the example given above is for the scenario where we want a particular branch to be copied into our local machine.

See something interesting. Clone it

Github is sort of the social network of developers where code does most of the talking. If we see a repo that we would like to hack in, we can get it in our local machine easily

git clone <'remote_repository_git_url'>

In case of github, this url can be seen on the right hand side of the repository page.

Final Thoughts

I have barely scratched the surface on the true potential of git and Github, it is something i believe that should be experienced by every developer. Github is really fuelling the opensource development community with tools that makes their work really easy. Incase you have never tried git or github before I strongly encourage you to give it a try.
